# personalweb Review Rubric

本项目完整交付前，必须由 3 个独立 AI subagent 分别评审成品、真实浏览器桌面/移动截图和代码。每个角色评分必须大于或等于 90，且不得出现红线。

## 共同红线

- 构建、类型检查、lint、测试或敏感信息扫描失败。
- 桌面或移动主路径不可用。
- 移动端出现明显遮挡、横向溢出、文字压出容器或关键内容不可读。
- 页面仍有明显脚手架、模板站、默认 favicon、默认文案或无意义装饰。
- AIFocus、CodePath 或个人身份展示事实不准确。
- 公开页面链接指向 404、私有仓库或尚未公开的资源。
- 页面、文档、截图、提交历史中泄露 API key、token、密码、本机私密路径或未公开配置。
- 3 个评审角色任一低于 90 分。

## 重度 Vibe Coder

关注“能不能让人相信这是一个真的在构建系统的人”。

- 首屏是否立即表达 `lgk-code`、AI 系统、产品工程和真实项目证据。
- AIFocus / CodePath 是否被展示成可运行系统，而不是泛泛作品列表。
- 文案是否有判断力、现场感和技术质地。
- 动效、图像和结构是否服务于记忆点，而不是装饰。
- 代码是否支撑这种表达，而不是靠硬编码堆字。

## 原生审美设计师

关注“是否像一个有审美自觉的人做出的个人网站”。

- 页面层次、留白、密度、字体尺寸和对比是否专业。
- 桌面和移动布局是否都成立，且没有模板化卡片堆叠感。
- 色彩是否有辨识度，并避免单一米色、紫蓝渐变或廉价科技感。
- 项目图片是否真实、有用、清晰，并且没有遮挡主要内容。
- 交互状态、focus 状态和 404 等边界页面是否被照顾。

## 破坏性质量官

关注“哪里会坏，哪里会泄露，哪里会误导”。

- `npm run quality` 必须通过，并说明最新一次运行结果。
- `robots.txt`、`sitemap.xml`、metadata、JSON-LD、favicon 和 Open Graph 图是否可用。
- 生成资产是否可复现，静态 HTML 检查是否能抓关键退化。
- 安全边界是否正确：只读相邻项目、不泄露秘密、不伪造部署域名。
- 对浏览器截图、audit 漏洞、依赖风险和已知限制是否诚实记录。

## 评分方式

- 每个角色输出 0-100 分和简短理由。
- 90-94：可交付，但仍有局部锐度或验证缺口。
- 95-97：高质量交付，只有轻微优化空间。
- 98-100：在当前约束下接近极致，且证据链完整。
- 若发现任一红线，直接判定不达标，不用平均分抵消。
